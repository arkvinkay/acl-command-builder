Dokumentasi Permission Command Builder

Halaman ini dirancang sebagai panduan pengguna sekaligus materi pembelajaran tentang permission/ACL pada Windows dan Linux. Fokusnya adalah memahami konsep dasar, memetakan opsi UI ke perintah yang dihasilkan, serta menerapkan praktik aman.

Peringatan penting
  • Perubahan permission/ACL dapat memutus akses aplikasi, user, atau sistem. Selalu uji di folder uji terlebih dahulu.
  • Sebelum perubahan skala besar, lakukan backup ACL (Windows: /save, Linux: getfacl -R).
  • Dokumentasi ini memakai istilah umum. Hasil akhir tetap dipengaruhi konteks (inheritance, policy, dan kondisi objek).

1) Konsep dasar yang wajib dipahami

Pembelajaran

1.1 Permission vs ACL vs Atribut

  • Permission (Linux classic): mode bit r/w/x untuk u/g/o.
  • ACL (Windows & Linux POSIX ACL): daftar entri izin untuk principal tertentu (user/group) dengan aturan evaluasi tertentu.
  • Atribut (Windows attrib read-only): bukan keamanan ACL. Atribut dapat memengaruhi perilaku aplikasi, tetapi tidak setara dengan kontrol akses NTFS.

1.2 Makna r/w/x pada file vs folder (Linux)

Bit File Folder/Directory
r Membaca isi file Melihat daftar nama file (list)
w Menulis/mengubah isi file Membuat/menghapus/rename entri di folder (umumnya perlu x juga)
x Menjalankan (execute) jika file executable Masuk/traverse ke folder dan mengakses item di dalamnya

Contoh umum: folder yang bisa “dibaca daftar file”-nya tetapi tidak bisa masuk ke dalam folder sering disebabkan r ada, namun x tidak ada.

1.3 Owner, group, dan “other” (Linux) serta ownership (Windows)

  • Linux memiliki konsep owner dan group; permission dasar selalu dihitung dari u/g/o serta ACL bila ada.
  • Windows memiliki owner objek (NTFS owner). Owner berhak mengubah ACL (tergantung kebijakan), sehingga ownership penting saat “reclaim access”.

1.4 Inheritance / pewarisan

  • Windows: inheritance diatur oleh flag ACE (OI/CI/IO/NP) dan state inheritance objek (/inheritance:e|d|r).
  • Linux: permission dasar tidak “inherit” seperti Windows. Pewarisan biasanya terjadi lewat:
    • Default ACL pada folder (POSIX ACL) — berlaku untuk item baru.
    • setgid pada folder — mempengaruhi group untuk item baru (tergantung FS).

1.5 Chmod numeric vs symbolic (Linux)

  • Numeric: 3 digit (atau 4 digit jika special bits) — contoh 755.
  • Symbolic: aturan eksplisit — contoh u=rwX,g=rX,o=, g+w.
Tambahan materi: special bits (setuid/setgid/sticky)

Special bits (digit ke-4 pada chmod numeric) sering digunakan pada sistem Linux.

Bit Angka Makna umum Contoh
setuid 4xxx Program berjalan dengan privilege owner file chmod 4755 /usr/bin/passwd
setgid 2xxx Pada folder: file baru cenderung mewarisi group folder chmod 2775 /shared
sticky 1xxx Pada folder: hanya owner file (atau root) yang dapat menghapus file-nya chmod 1777 /tmp

Gunakan special bits dengan sangat hati-hati. Pada lingkungan produksi, ikuti kebijakan keamanan organisasi.

2) Panduan UI: pemetaan opsi ke perilaku generator

Panduan pengguna

2.1 OS

  • Windows menghasilkan perintah berbasis icacls, takeown, dan attrib.
  • Linux menghasilkan perintah berbasis chmod, chown/chgrp, setfacl/getfacl.

2.2 Path file/folder

  • Generator akan menambahkan tanda kutip bila path mengandung spasi.
  • Di Windows, format umum: C:\Folder\File.txt. Di Linux: /path/target.

2.3 Tipe target

  • Otomatis: membantu preset Linux “folder 755, file 644” jika kamu memilih preset otomatis.
  • File / Folder: memaksa interpretasi saat memakai preset otomatis.

2.4 Mode

  • Dasar: fokus pada opsi inti.
  • Lanjutan: menampilkan blok opsi tambahan (workflow / ACL).
  • Pakar: membuka input argumen manual yang akan ditempel ke perintah (gunakan hanya jika benar-benar paham).

2.5 Rekursif

  • Windows: menggunakan /t pada icacls untuk menerapkan perubahan ke isi folder.
  • Linux:
    • Untuk chmod/chown/chgrp, bisa memakai -R.
    • Jika memilih “Hanya file” atau “Hanya folder”, generator membungkus dengan find -type f/d.

2.6 Format output

  • Multi-baris: setiap command di baris terpisah (lebih mudah dibaca & audit).
  • Satu baris (&&): command dirangkai dengan operator && (lebih ringkas, tetapi lebih sulit dibaca).

2.7 Tombol Salin

Menyalin output ke clipboard. Disarankan untuk menempelkan perintah ke catatan tiket/incident agar ada audit trail.

3) Windows: icacls / takeown / attrib (materi lengkap)

Windows

3.1 Gambaran umum model akses Windows (NTFS ACL)

  • Windows memakai ACL (Access Control List) yang berisi ACE (Access Control Entry).
  • Setiap ACE mengikat principal (user/group) dengan hak akses dan (opsional) flag inheritance.
  • Hak akses efektif dipengaruhi oleh: ACE explicit vs inherited, deny vs allow, membership group, dan scope inheritance.
3.2 Aksi Windows: kapan dipakai?
Aksi Tujuan Kapan digunakan
Tampilkan ACL (icacls) Melihat izin saat ini Selalu lakukan sebelum mengubah, sebagai baseline
/grant Menambahkan ACE allow Jika ingin menambah akses tanpa mengubah ACE principal lain
/grant:r Mengganti ACE untuk principal target Jika ingin hasil deterministik untuk principal tertentu
/deny Menambahkan ACE deny Jika memang wajib menolak akses (hati-hati)
/remove Menghapus ACE principal Jika ingin membersihkan izin khusus
/reset Reset ACL ke default inheritance Jika ACL “berantakan” dan ingin kembali ke baseline folder parent
/inheritance:e|d|r Atur pewarisan dari parent Jika perlu memutus/menormalkan inheritance
/setowner Ubah owner Jika ownership salah dan menghalangi pengelolaan ACL
/save /restore Backup & restore ACL Sebelum operasi besar; atau rollback cepat
takeown Reclaim ownership Jika akses tertutup dan perlu “ambil alih” kepemilikan
attrib +/-R Atur atribut Read-only Untuk kebutuhan aplikasi/atribut, bukan kontrol keamanan ACL

3.3 Principal (user/group)

  • Principal dapat berupa user lokal, domain user, atau group.
  • Format umum: DOMAIN\User, User, atau group seperti Users.
  • Efektif permission dipengaruhi membership group. Jika user anggota beberapa group, izin bisa “bertambah”.

3.4 Hak akses standar (icacls)

Kode Nama Catatan
F Full control Mencakup hampir semua hak, termasuk mengubah permission/owner (tergantung konteks)
M Modify Umumnya untuk edit konten + delete, tanpa set permission/owner
RX Read & execute Membaca dan menjalankan/men-traverse
R Read Read-only
W Write Tulis/ubah konten (dengan batasan tertentu)
D Delete Hak delete (sering dianggap “lanjutan” oleh banyak admin)
3.5 Hak akses granular (granular rights) — arti dan kapan digunakan

Builder menyediakan opsi granular untuk skenario spesifik. Tidak semua hak relevan untuk semua tipe objek.

Kode Arti ringkas Umum dipakai untuk
RC Read control Membaca ACL/owner (bagian kontrol)
WDAC Write DAC Mengubah permission (ACL). Sangat sensitif.
WO Write owner Mengubah owner. Sangat sensitif.
S Synchronize Sinkronisasi handle; biasanya ikut dalam banyak set hak
RD Read data / List directory File: baca data; Folder: list directory
WD Write data / Add file File: tulis data; Folder: tambah file
AD Append data / Add subdirectory File: append; Folder: tambah subfolder
REA Read extended attributes Baca EA
WEA Write extended attributes Tulis EA
X Execute / Traverse File: execute; Folder: traverse
DC Delete child Folder: menghapus child item (tergantung konteks)
RA Read attributes Membaca atribut
WA Write attributes Mengubah atribut
DE Delete Hak delete eksplisit
GR Generic read Set generik (OS map ke hak-hak dasar)
GW Generic write Set generik
GE Generic execute Set generik
GA Generic all Set generik “all”
Pedoman praktis
  • Jika tujuanmu “user bisa mengubah izin”, itu biasanya melibatkan WDAC (berisiko tinggi).
  • Jika tujuanmu “user bisa menghapus file/folder”, pastikan hak delete sesuai kebutuhan (mis. D/DE), dan pahami dampaknya pada child.

3.6 Inheritance: OI / CI / IO / NP

Flag inheritance adalah bagian dari ACE (bukan switch global). Berikut interpretasi yang paling berguna untuk praktik.

Flag Makna Contoh penggunaan
(OI) Object inherit — diwariskan ke file Memberi izin ke file-file di bawah folder
(CI) Container inherit — diwariskan ke folder Memberi izin ke subfolder
(IO) Inherit only — tidak berlaku pada objek ini, hanya pada turunan “Hanya turunan” (inherit-only policy)
(NP) No propagate — berhenti di satu level turunan Memberi izin hanya ke anak langsung, bukan cucu

3.7 Pemetaan “Ruang lingkup” builder ke flag inheritance

Ruang lingkup (UI) Tujuan Flag yang lazim
Hanya objek ini ACE berlaku hanya pada target (tanpa OI/CI/IO/NP)
Objek ini + turunan Berlaku pada folder ini dan diwariskan ke bawah Umumnya (OI)(CI)
Hanya turunan Tidak berlaku pada folder ini, hanya diwariskan (IO) + (OI/CI) sesuai target

3.8 /inheritance:e|d|r — perbedaan yang sering membingungkan

Mode Arti ringkas Dampak umum
/inheritance:e Enable inheritance Objek menerima inherited ACE dari parent
/inheritance:d Disable inheritance (tetap simpan izin yang ada) Pewarisan berhenti; inherited ACE umumnya menjadi explicit pada objek ini
/inheritance:r Remove inherited ACE Menghapus entri inherited; berisiko memutus akses jika explicit ACE tidak cukup
Rekomendasi
  • Gunakan /inheritance:r hanya jika kamu paham baseline explicit ACE yang dibutuhkan.
  • Sebelum mengubah inheritance, lakukan backup ACL.

3.9 /grant vs /grant:r dan “deny” precedence (pemahaman praktis)

  • /grant menambahkan ACE. Jika principal sudah punya ACE lain, hasil efektif bisa jadi “bertumpuk”.
  • /grant:r mengganti ACE untuk principal target, sehingga lebih deterministik.
  • /deny perlu kehati-hatian: deny dapat mengalahkan allow dalam banyak kasus.
Ringkasan evaluasi (simplifikasi yang aman)
  • ACE explicit umumnya dievaluasi sebelum ACE inherited.
  • Deny biasanya diproses sebelum allow pada level yang sama.
  • Karena membership group, deny pada salah satu group dapat memengaruhi user.

Evaluasi lengkap Windows cukup kompleks. Untuk praktik, prinsip aman: minimalkan deny, gunakan allow yang jelas, dan verifikasi dengan uji akses / icacls output sebelum dan sesudah.

3.10 Backup & restore ACL (best practice)

Gunakan pola berikut sebelum perubahan besar pada folder: 

REM Backup ACL
icacls "C:\TargetFolder\*" /save "C:\temp\acl-backup.txt" /t

REM Restore ACL (jalankan pada root yang sama)
icacls "C:\TargetFolder" /restore "C:\temp\acl-backup.txt"
  • Save biasanya memakai wildcard \* agar mencakup isi.
  • Restore dilakukan pada root folder yang konsisten dengan struktur saat backup.
  • Untuk rollback cepat, simpan file backup bersama tiket/incident.

3.11 takeown vs icacls /setowner

  • takeown berguna untuk “mengambil alih” kepemilikan saat akses terkunci.
  • icacls /setowner menetapkan owner secara eksplisit (butuh privilege yang memadai).
  • Jika tujuanmu hanya mengubah ACL, terkadang ownership bukan masalah. Namun pada kasus tertentu, ownership menjadi penghalang.

3.12 attrib read-only bukan kontrol keamanan

  • attrib +R menandai file/folder sebagai read-only (atribut).
  • ACL NTFS tetap menjadi mekanisme keamanan utama. Atribut read-only tidak menggantikan deny/allow ACL.

3.13 Verifikasi & audit (Windows)

  • Lihat ACL: icacls "C:\path"
  • Cek group user (bantuan analisis efektif permission): whoami /groups
  • Catat perubahan: simpan output command + snapshot icacls sebelum/sesudah.

4) Linux: chmod / chown / setfacl (materi lengkap)

Linux

4.1 chmod numeric (angka) — cara menghitung

Setiap kategori (u/g/o) memakai bobot: r=4, w=2, x=1.

Nilai Bit Makna
7 rwx Read + write + execute
6 rw- Read + write
5 r-x Read + execute
4 r-- Read
0 --- Tanpa izin

4.2 chmod symbolic (huruf) — operator +, -, =

  • u owner, g group, o other, a all.
  • + menambah permission, - menghapus, = menetapkan tepat seperti yang disebut. 
# Tambah write untuk group
chmod g+w /path/file

# Set persis: owner=rw, group=r, other=
chmod u=rw,g=r,o= /path/file

# Contoh builder: gunakan X agar rekursif lebih aman
chmod -R u=rwX,g=rX,o= /path/dir

4.3 “X” (kapital) vs “x” saat rekursif

  • x: selalu memberi execute (berisiko membuat semua file menjadi executable).
  • X: memberi execute hanya untuk folder atau file yang sebelumnya sudah executable — lebih aman untuk operasi rekursif.

4.4 Preset umum: 755/644 dan alasan praktisnya

  • Folder 755: owner rwx, group r-x, other r-x — bisa traverse.
  • File 644: owner rw-, group r--, other r-- — read-only untuk non-owner.
Tambahan materi: umask (mengapa default permission sering tidak “penuh”)

umask memotong izin default saat file/folder dibuat. Karena itu, “default” sering terlihat seperti 644/755. Ini bukan hasil chmod, melainkan hasil kebijakan penciptaan file + umask.

4.5 chown / chgrp

  • chown user:group target mengubah owner dan group.
  • chgrp group target mengubah group saja.
  • Operasi rekursif memakai -R atau dibungkus find bila perlu seleksi file/folder saja.

4.6 POSIX ACL: setfacl / getfacl

ACL dipakai saat permission dasar tidak cukup (misalnya memberi akses spesifik kepada 1 user tanpa mengubah group).

4.6.1 Struktur entri ACL

  • User tertentu: u:alice:rwX
  • Group tertentu: g:staff:r-X
  • Mask: m::rwx (membatasi effective untuk entri user/group)
  • Other: o::r--
4.6.2 Mask ACL: sumber “izin sudah diset tapi tidak efektif”

Pada banyak sistem, mask membatasi izin efektif untuk entri user/group ACL. Jadi meskipun kamu set u:alice:rwx, jika mask m::rw-, izin efektif alice menjadi rw-. 

# Lihat ACL dan "effective"
getfacl /path/target

# Set entri user + set mask agar sesuai kebutuhan
setfacl -m u:alice:rwx,m::rwx /path/target

4.6.3 Default ACL (folder) vs objek existing

  • Default ACL (prefix d:) hanya mempengaruhi objek baru yang dibuat di bawah folder.
  • Untuk objek existing, kamu perlu operasi rekursif (mis. setfacl -R atau find).

4.7 Backup & restore ACL (Linux)

# Backup ACL rekursif
sudo getfacl -R '/path/target' > './acl-backup.txt'

# Restore ACL (membaca file backup)
sudo setfacl --restore='./acl-backup.txt'

4.8 Verifikasi & audit (Linux)

  • Lihat permission: ls -l dan stat file
  • Lihat ACL: getfacl path
  • Lihat ownership: ls -ld dir
  • Jika hasil tidak sesuai, cek mask ACL dan apakah default ACL diterapkan pada objek existing atau tidak.

5) Cookbook: skenario umum (siap pakai + penjelasan)

Praktik

Bagian ini berisi resep cepat. Gunakan sebagai referensi saat membuat perintah lewat builder, lalu verifikasi hasilnya.

5.1 Windows — memberi Modify untuk user pada folder + seluruh isi

icacls "C:\TargetFolder" /grant "DOMAIN\User":(OI)(CI)(M) /t
  • (OI)(CI) = diwariskan ke file dan folder.
  • /t = menerapkan ke isi folder yang sudah ada.

5.2 Windows — izin hanya untuk turunan (inherit-only)

icacls "C:\TargetFolder" /grant "DOMAIN\User":(OI)(CI)(IO)(RX)

5.3 Windows — memutus inheritance tanpa menghapus izin yang ada

icacls "C:\TargetFolder" /inheritance:d

5.4 Windows — backup ACL sebelum perubahan besar

icacls "C:\TargetFolder\*" /save "C:\temp\acl-backup.txt" /t

5.5 Linux — folder 755 dan file 644 (rekursif, dipisah)

sudo find '/path/target' -type d -exec sudo chmod 755 {} +
sudo find '/path/target' -type f -exec sudo chmod 644 {} +

5.6 Linux — memberi akses spesifik ke satu user tanpa ubah group

sudo setfacl -m u:alice:rwX '/path/target'

5.7 Linux — menetapkan default ACL agar item baru otomatis punya akses

sudo setfacl -m d:u:alice:rwX '/path/shared'

Default ACL mempengaruhi item baru, bukan yang sudah ada.

5.8 Linux — hapus ACL tambahan dan kembali ke mode bit

sudo setfacl -b '/path/target'

6) Troubleshooting: ketika hasil tidak sesuai ekspektasi

Diagnostik

6.1 Windows: gejala umum

  • Akses masih ditolak setelah grant:
    • Periksa apakah ada deny pada user atau salah satu group user.
    • Periksa apakah ACE yang kamu tambah hanya inherited-only (IO) sehingga tidak berlaku pada objek target.
    • Verifikasi membership: whoami /groups
  • Perubahan tidak menyentuh isi folder:
    • Pastikan /t aktif untuk menerapkan ke item existing.
  • Ingin “rollback cepat”:
    • Gunakan file backup ACL (/restore). 
icacls "C:\TargetFolder"
whoami /groups

6.2 Linux: gejala umum

  • Sudah setfacl tetapi tidak efektif:
    • Cek mask ACL (getfacl menampilkan “effective”).
    • Sesuaikan mask jika diperlukan (setfacl -m m::rwx ...).
  • Default ACL tidak mempengaruhi file lama:
    • Default ACL hanya untuk objek baru; gunakan rekursif untuk existing.
  • Folder bisa list tapi tidak bisa akses file:
    • Biasanya karena x (traverse) tidak ada pada folder. 
ls -ld /path/dir
getfacl /path/dir
Checklist aman sebelum eksekusi
  1. Snapshot kondisi awal (Windows: icacls, Linux: ls -l + getfacl).
  2. Backup ACL jika skala besar.
  3. Pastikan scope benar (this vs children vs children-only).
  4. Uji pada 1 folder kecil terlebih dahulu.
  5. Catat command yang dipakai sebagai evidence.

Jika kamu menambah opsi baru di builder, perbarui dokumentasi ini agar pemetaan UI → command tetap konsisten.
